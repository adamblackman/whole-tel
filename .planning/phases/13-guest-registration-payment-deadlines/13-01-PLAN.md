  <name>Task 1: Create database migration for guest registration and deadlines</name>
  <files>supabase/migrations/20260324000001_guest_registration_deadlines.sql</files>
  <action>
Create a single migration file that does the following in order:

1. **Widen bookings status CHECK constraint** to include 'expired':
   - `ALTER TABLE bookings DROP CONSTRAINT IF EXISTS bookings_status_check;`
   - `ALTER TABLE bookings ADD CONSTRAINT bookings_status_check CHECK (status IN ('pending', 'confirmed', 'cancelled', 'expired'));`

2. **Add deadline columns to bookings**:
   - `payment_deadline timestamptz` (36 hours after creation)
   - `activity_deadline timestamptz` (LEAST of check_in - 30 days or created_at + 7 days)
   - `stripe_checkout_url text` (stored at Stripe session creation for re-access)

3. **Backfill existing bookings** (for any pending bookings that already exist):
   - `UPDATE bookings SET payment_deadline = created_at + interval '36 hours', activity_deadline = LEAST(check_in::timestamptz - interval '30 days', created_at + interval '7 days') WHERE payment_deadline IS NULL;`

4. **Add registration columns to booking_invitations**:
   - `full_name text`
   - `phone text`

5. **Create atomic increment RPC function** `increment_booking_guest_count(p_booking_id uuid)`:
   - Returns `int` (new guest_count)
   - `LANGUAGE sql SECURITY DEFINER`
   - Atomically increments `guest_count` by 1 and sets `updated_at = now()`
   - Only for bookings with `status IN ('pending', 'confirmed')`
   - Guards against exceeding `max_guests` from the properties table via a subquery
   - `RETURNING guest_count`

Note: The GiST exclusion index already uses `WHERE (status = 'confirmed')`, so expired bookings are automatically excluded from date-range blocking. No index changes needed.
