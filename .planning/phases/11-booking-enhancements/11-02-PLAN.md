       - Use same inline style objects pattern as booking-confirmed.tsx (body, container, heading, paragraph, hr, footer)

    2. Create `src/lib/actions/booking-invitations.ts` with three Server Actions:

       **`sendInvitation(input: { bookingId: string; email: string })`:**
       - `'use server'` directive
       - `verifySession()` to get user
       - Validate with `sendInvitationSchema`
       - Fetch booking via supabase `.eq('id', bookingId).eq('guest_id', user.id).single()` -- must own the booking
       - Also fetch property name, location from the booking's property join
       - Fetch inviter's profile for `display_name`
       - Check for existing invitation with same (booking_id, email): if exists and status is 'pending', resend email; if 'accepted', return "already accepted"; if 'declined', allow re-invite (update status back to 'pending')
       - Insert into `booking_invitations`: `{ booking_id, email, invited_by: user.id }`
       - Derive `acceptUrl` from `process.env.NEXT_PUBLIC_APP_URL` + `/bookings/invitations/{token}` (token from the inserted row -- select it back)
       - Send email via `getResend().emails.send()` with `BookingInvitationEmail` rendered via `react` option (same pattern as webhook confirmation email)
       - `revalidatePath('/bookings')`
       - Return `{ success: true }`

       **`acceptInvitation(token: string)`:**
       - `'use server'` directive
       - `verifySession()` to get user
       - Fetch user's profile email from supabase `profiles` table
       - Use `createAdminClient()` to fetch invitation by token (RLS may not cover token-based lookup for all users)
       - Validate: invitation exists, status is 'pending', email matches user's profile email
       - If email doesn't match, return `{ error: 'This invitation was sent to a different email address' }`
       - Update invitation: `{ status: 'accepted', accepted_by: user.id, updated_at: new Date().toISOString() }`
       - Increment guest count: fetch current booking guest_count, call the `updateGuestCount` pattern from `booking-updates.ts` (reuse the recalculation logic -- either call the exported function or duplicate the admin update pattern). Since `updateGuestCount` validates ownership (guest_id === user.id) and the acceptor is NOT the booking owner, create a separate internal helper or use `createAdminClient()` directly here to increment guest_count and recalculate.
         * Fetch booking + property + booking_add_ons
         * Call `calculatePricing()` with guest_count + 1
         * Update booking via admin client with new totals and incremented guest_count
       - `revalidatePath('/bookings')`
       - Return `{ success: true, bookingId: invitation.booking_id }`

       **`declineInvitation(token: string)`:**
       - `'use server'` directive
       - `verifySession()` to get user
       - Fetch user's profile email
       - Use `createAdminClient()` to fetch invitation by token
       - Validate: invitation exists, status is 'pending', email matches
       - Update invitation: `{ status: 'declined', updated_at: new Date().toISOString() }`
       - `revalidatePath('/bookings')`
       - Return `{ success: true }`

    IMPORTANT: Use `createAdminClient()` (not regular client) for invitation operations that cross user boundaries (the invited user updating the booking owner's booking totals). The Server Action validates auth via `verifySession()` and validates email match before any update.

    IMPORTANT: For email rendering, use `import { render } from '@react-email/render'` if needed, or pass the component directly to Resend's `react` option (Resend supports React components natively).
