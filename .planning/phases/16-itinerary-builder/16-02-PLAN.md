2. Create `src/lib/actions/itinerary.ts` with 'use server' directive:

   First, export two pure validation helper functions (these are unit-testable without Supabase):

   `isEventDateInRange(eventDate: string, checkIn: string, checkOut: string): boolean`
   - Returns true if eventDate >= checkIn AND eventDate < checkOut
   - Events are rejected on checkout day because guests depart on that day (checkout day is exclusive upper bound)

   `isDeadlinePassed(activityDeadline: string | null, now?: Date): boolean`
   - Returns false if activityDeadline is null (no deadline set)
   - Returns true if (now ?? new Date()) > new Date(activityDeadline)
   - The optional `now` parameter allows deterministic testing

   Then the Server Actions that use these helpers:

   `upsertItineraryEvent(bookingId: string, payload: ItineraryEventInput): Promise<{ error?: string; id?: string }>`
   - verifySession() from dal.ts
   - Parse payload with ItineraryEventSchema
   - Query booking: select id, check_in, check_out, activity_deadline from bookings WHERE id = bookingId AND guest_id = user.id (RLS also protects but explicit check gives better errors)
   - If !booking, return { error: 'Booking not found.' }
   - Activity deadline enforcement using isDeadlinePassed(booking.activity_deadline): if true, return { error: 'Activity booking deadline has passed.' }
   - Date range validation using isEventDateInRange(payload.eventDate, booking.check_in, booking.check_out): if false, return { error: 'Event date must be within booking dates (checkout day excluded — guests depart that day).' }
   - If payload has an id field (update case): use that id. Otherwise generate with crypto.randomUUID() — but actually, let the client pass `id` (generated client-side via crypto.randomUUID()) for upsert pattern. Accept optional `id?: string` in the function params.
   - Supabase .upsert() with: id (passed or gen_random_uuid via DB default), booking_id, activity_id, title, event_date, start_time (append ':00' for TIME format if only HH:MM), end_time, notes
   - revalidatePath(`/bookings/${bookingId}/itinerary`)
   - Return { id: data.id }

   `deleteItineraryEvent(bookingId: string, eventId: string): Promise<{ error?: string }>`
   - verifySession()
   - Query booking for deadline check (same as above)
   - Activity deadline enforcement using isDeadlinePassed
   - Delete from itinerary_events WHERE id = eventId (RLS ensures guest ownership)
   - revalidatePath
   - Return {}

TIMEZONE CONVERSION (critical — Research Pitfall 3):
- Add two helper functions at the top of the file (from Research Pattern 5):
  ```typescript
  function toDateStr(d: Date, tz: string): string {
    return new Intl.DateTimeFormat('en-CA', { timeZone: tz, year: 'numeric', month: '2-digit', day: '2-digit' }).format(d)
  }
  function toTimeStr(d: Date, tz: string): string {
    return new Intl.DateTimeFormat('en-US', { timeZone: tz, hour: '2-digit', minute: '2-digit', hour12: false }).format(d)
  }
  ```
- When handling dateClick: FullCalendar's DateClickArg.date is a JavaScript Date in UTC. You MUST convert it to wall-clock HH:MM using `toTimeStr(arg.date, timezone)` and the date using `toDateStr(arg.date, timezone)` before passing to AddEventDialog or ActivityPicker as pre-filled values. Without this conversion, a guest in PST clicking 9:30 AM would see 17:30 pre-filled (the UTC equivalent), and the stored start_time would be wrong.
- Pass the `timezone` prop down to both AddEventDialog and ActivityPicker so they can display times in the property's timezone context.

- On dateClick (from interactionPlugin): if not isLocked, convert arg.date using toDateStr/toTimeStr with timezone, then open AddEventDialog with the converted date and time pre-filled
- On eventClick: if not isLocked, show a confirmation to delete the event. On confirm, call deleteItineraryEvent Server Action and remove from local state.
- Auto-save pattern: when an event is added (from either dialog), call upsertItineraryEvent via useTransition with 400ms debounce (see research Pattern 4). Add event to local state immediately for optimistic UI, then save server-side.
- Show a subtle "Saving..." indicator when isPending is true (small text or spinner near the header)
- If isLocked, disable dateClick handler and show events as read-only
